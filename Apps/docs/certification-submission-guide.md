## <a name="introduction"></a>Введение

В рамках программы соответствия требованиям приложений Microsoft 365 сертификация Microsoft 365 обеспечивает гарантии и уверенность корпоративных организаций в том, что данные и конфиденциальность должным образом защищены и защищены при интеграции сторонних приложений и надстроек для разработчиков на платформу Microsoft 365. Приложения и надстройки, которые проходят проверку, будут назначаться сертификацией **Microsoft 365** в экосистеме Microsoft 365. 

Принимая участие в программе сертификации Microsoft 365, вы соглашаетесь с этими дополнительными условиями и соответствуете любой сопроводительной документации, которая относится к вашему участию в программе сертификации Microsoft 365 с корпорацией Майкрософт ("Майкрософт", "мы", "мы" или "наш"). Вы представляете и гарантируете нам, что у вас есть право принимать эти дополнительные условия сертификации Microsoft 365 от имени себя, компании и (или) другой сущности, если это применимо. Мы можем изменить, изменить или завершить эти дополнительные термины в любое время. Постоянное участие в программе сертификации Microsoft 365 после любого изменения или изменения означает, что вы согласны с новыми дополнительными условиями. Если вы не согласны с новыми дополнительными условиями или если мы прекращаем эти дополнительные условия, необходимо прекратить участие в программе сертификации Microsoft 365.

Этот документ предназначен для независимых поставщиков программного обеспечения, чтобы предоставить сведения о процессе сертификации Microsoft 365, предварительные требования для запуска процесса и сведения о конкретных элементах управления безопасностью, которые должны иметь независимые поставщики программного обеспечения.  Общие сведения о программе соответствия требованиям приложений Microsoft 365 можно найти на странице программы соответствия требованиям приложений Microsoft 365[.](../overview.md) 

> [!IMPORTANT]
> В настоящее время сертификация Microsoft 365 применима ко всем:
>* Приложения Microsoft Teams (вкладки, боты и т. д.).
>* Приложения и надстройки Sharepoint
>* Надстройки Office (Word, Excel, PowerPoint, Outlook, Project, OneNote)
>* Веб-приложения

## <a name="prerequisites"></a>Предварительные требования

### <a name="publisher-attestation"></a>Аттестация издателя

Перед сертификацией Microsoft 365 необходимо завершить аттестацию издателя. Однако вы можете начать процесс сертификации Microsoft 365 до завершения аттестации издателя.  

### <a name="read-the-microsoft-365-certification-specification"></a>Ознакомьтесь со спецификацией сертификации Microsoft 365

Корпорация Майкрософт рекомендует всем независимым поставщикам программного обеспечения полностью ознакомиться с этой спецификацией сертификации Microsoft 365, чтобы обеспечить соответствие всех применимых элементов управления среде в области действия и приложению или надстройке. Это поможет обеспечить плавный процесс оценки.

## <a name="microsoft-365-certification-specification-updates"></a>Спецификация сертификации Microsoft 365 Актуальные данные 

Актуальные данные спецификации сертификации Microsoft 365 ожидается примерно каждые 6–12 месяцев. Эти обновления могут привести к новым целевым доменам безопасности и (или) элементам управления безопасностью. Актуальные данные будут основаны на отзывах разработчиков, изменениях в среде угроз и повышении базовых показателей безопасности программы по мере ее развития. 

Независимые поставщики программного обеспечения, которые уже начали оценку сертификации Microsoft 365, могут продолжить оценку с помощью версии спецификации сертификации Microsoft 365, которая была действительной при запуске оценки. Все новые отправки, включая годовую повторную сертификацию, должны быть оценены с учетом опубликованной версии.

> [!NOTE]
> Вам не нужно соответствовать всем элементам управления в этой спецификации сертификации Microsoft 365, чтобы получить сертификацию. Однако передача пороговых значений (которые не будут раскрыты) выполняется для каждого домена безопасности, рассматриваемого в этой спецификации сертификации Microsoft 365. Некоторые элементы управления будут классифицируются **как "** Жесткий сбой", что означает, что отсутствие этих элементов управления безопасностью приведет к сбою оценки. 

## <a name="certification-scope"></a>Область сертификации

Среда **в области** — это среда, которая поддерживает доставку кода приложения или надстройки и поддерживает все серверные системы, с которых может взаимодействовать приложение или надстройка. Любые дополнительные подключенные среды также будут включены в область, если не установлена надлежащая сегментация и подключенные среды не могут повлиять на безопасность среды в области. Все среды аварийного восстановления также должны быть включены в область оценки, так как эти среды потребуются для создания службы в случае возникновения каких-либо действий с основной средой.  Термин " **системные компоненты в области"** ссылаются на **все** устройства и системы, используемые в среде в области действия. В области компоненты включают в себя, но не ограничиваются:
* Веб-приложения.
* Серверов.
* Брандмауэры (или эквивалентные).
* Переключатели.
* Подсистемы балансировки нагрузки.
* Виртуальная инфраструктура.
* Порталы управления веб-сайтами поставщиков облачных служб 

> [!IMPORTANT]
> Среда в области действия должна иметь сеть периметра, а вспомогательная среда приложения или надстройки должна быть сегментировано от внутренних бизнес-систем и корпоративных сред, что ограничивает область действий оценки только в системах в области действия. Аналитики сертификации проверяют методы сегментации во время оценки, а также просматривают отчеты о тестировании на проникновение, которые должны включать тестирование для проверки эффективности любых используемых методов сегментации.

### <a name="infrastructure-as-a-service-iaas-platform-as-a-service-paas-and-software-as-a-service-saas"></a>Инфраструктура как услуга (IaaS), платформа как услуга (PaaS) и программное обеспечение как услуга (SaaS) 
Если IaaS и (или) PaaS используются для поддержки инфраструктуры приложения или доставки кода надстройки, поставщик облачной платформы будет отвечать за некоторые элементы управления безопасностью, оцениваемые в процессе сертификации. Таким образом, аналитикам сертификации необходимо предоставить независимые внешние рекомендации по обеспечению безопасности от поставщика облачной платформы с помощью внешних отчетов о соответствии, таких как [PCI DSS] Аттестация соответствия (AOC), ISO27001 или [SOC 2] Отчеты типа II. 

Приложение F содержит сведения о том, какие элементы управления безопасностью, скорее всего, будут применяться на основе следующих типов развертывания и в зависимости от того, будет ли приложение или надстройка exfiltrates данных M365 или нет: 
* ISV Hosted 
* Размещенная среда IaaS 
* PaaS/Serverless Hosted 
* Гибридное размещение 
* Размещение с общим доступом 

При развертывании IaaS или PaaS необходимо предоставить свидетельство среды, размещенной в этих типах развертывания.

### <a name="sampling"></a>Выборки

Запросы на подтверждение в поддержке оценки сертификации должны быть основаны на примере системных компонентов в области с учетом различных операционных систем, основной функции устройства и различных типов устройств. Подходящий пример будет выбран в начале процесса сертификации. В качестве руководства по выборке размера можно использовать следующую таблицу:

|Размер генеральной совокупности              | Пример                  |
|---------------------------- |-------------------------|
|<5|1|
|>5 & <10|2|
|>9 & <25|3|
|>24|4|

> [!NOTE]
>Если между устройствами, включенными в начальный пример, выявлены несоответствия, размер выборки может быть увеличен во время оценки. 

## <a name="certification-process"></a>Процесс сертификации

Перед началом процесса сертификации необходимо успешно завершить аттестацию издателя. После завершения процесса сертификации Microsoft 365 будет выполняться следующее:

### <a name="preparation"></a>Подготовка
1. Перейдите в Центр партнеров и просмотрите документацию по завершенной [аттестации издателя](../docs/attestation.md) . При необходимости вы можете изменять и обновлять ответы. Однако в этом случае вам потребуется повторно отправить документацию по аттестации для утверждения. Если отправка старше трех месяцев, потребуется повторно отправить аттестацию издателя для проверки и проверки. 
1. Внимательно прочтите руководство по отправке сертификации [Microsoft 365](../docs/certification-submission-guide.md) , чтобы понять, что вам потребуется. Убедитесь, что вы сможете выполнить требования к элементам управления [, указанные](../docs/certification-submission-guide.md#app-certification-criteria) в руководстве по отправке сертификации Microsoft 365.
1. В Центре партнеров щелкните "Начать сертификацию". Откроется начальный портал отправки документов. Отправьте [начальную отправку документа](../docs/certification-submission-guide.md#initial-document-submission). Это поможет нам определить, что находится в области оценки, в зависимости от того, как ваше приложение спроектировано и обрабатывает данные клиента. Часто проверяйте эту страницу, чтобы узнать, была ли ваша отправка принятой.

>[!NOTE]
>Для всех приложений Office вы можете ссылаться на наше руководство [пользователя по приложениям Office](../docs/userguide.md). Для всех веб-приложений можно ссылаться на наше [руководство пользователя по приложениям SaaS](../docs/saasuserguide.md).

### <a name="assessment"></a>Оценка
1. После принятия первоначальной отправки документа на портале автоматически отобразится набор элементов управления безопасностью, необходимых для вашего приложения. Затем вам потребуется отправить свидетельство для каждого элемента управления, демонстрирующее, что элемент управления установлен. Помните, что вам будет **предоставлено 60 дней для** отправки всех свидетельств. Аналитик проверит ваше свидетельство и утвердит элемент управления или запросит новое или дополнительное свидетельство. Часто проверяйте эту страницу, чтобы узнать, принято ли ваше свидетельство.
### <a name="certification"></a>Сертификация
1. После проверки отправки аналитиком вы будете уведомлены о решении о сертификации. Приложения, отмеченные сертификацией, получат эмблему на своем приложении в **AppSource** и на страницах **документации** Майкрософт. Здесь вы можете узнать о всех преимуществах [сертификации](../docs/enterprise-app-certification-guide.md#program-benefits).

## <a name="review-and-re-certification"></a>Проверка и повторная сертификация
В случае, если ваше приложение в любой [](../docs/certification-submission-guide.md#significant-changes) момент преобразует значительные изменения, вам потребуется уведомить нас.

Вам также потребуется проходить повторную сертификацию на ежегодной основе. Для этого потребуется повторная проверка элементов управления в области в текущей среде. Этот процесс может начаться до 90 дней до истечения срока действия сертификации. Срок действия существующей сертификации не истечет в течение периода повторной сертификации. Срок действия повторной сертификации для всех программ истекает в год с годовщиной сертификации Microsoft 365.

Если сертификация не продлена до даты окончания срока действия, состояние сертификации приложений будет отозвано. Все компоненты, значки и связанная с ним фирменная символика сертификации будут удалены из приложения, и вам будет запрещено рекламировать приложение как сертифицированное для Microsoft 365.


> [!IMPORTANT]
> **Интервал времени отправки:** Предполагается, что в среднем процесс оценки должен занять 30 дней, если вы можете часто проверять состояние отправки и своевременно реагировать на комментарии и дополнительные запросы на свидетельство. После запуска процесса сертификации для завершения оценки разрешается не более 60 дней. Если все отправки не были завершены в течение 60-дневного периода времени, отправка завершится ошибкой и процесс должен начаться заново. Эти результаты не будут общедоступными.

|**Отчет о тестировании на проникновение** |Отчет о тестировании на проникновение, завершенный в течение последних 12 месяцев. Укажите, какой тип среды размещения используется. IsV Hosted, IaaS, PaaS или Hybrid. **Примечание:** Если используется SaaS, укажите различные службы SaaS, которые используются для предоставления вспомогательных служб в среде.|
|**Общедоступная занимаемая память** | Подробное **описание всех** общедоступных IP-адресов и URL-адресов, используемых вспомогательной инфраструктурой. К ним должен относится полный диапазон маршрутизируемых IP-адресов, выделенный для среды, если не реализована надлежащая сегментация для разделения используемого диапазона (требуется соответствующее свидетельство сегментации).|
|**С помощью схем потока данных** |Схемы потоков со следующими сведениями:
||&#x2713; данных M365 в приложение или надстройку и из нее (включая [EUII](#euii) и [OII](#oii) ).|
||&#x2713; потоки данных M365 в вспомогательной инфраструктуре (если применимо)|
||&#x2713; схемы, на которых показано, где и какие данные хранятся, как данные передаются внешним третьим лицам (включая сведения о третьих лицах) и как данные защищаются при передаче через открытые или общедоступные сети и неактивные данные.|

|**Разрешения API Майкрософт**| Предоставьте документацию, в которой подробно описаны все используемые API Майкрософт, а также разрешения, запрашиваемые для работы приложения или надстройки, а также обоснование запрашиваемых разрешений.|

## <a name="evidence-collection-and-assessment-activities"></a>Действия по сбору и оценке свидетельств

Аналитикам сертификации потребуется проверить свидетельство во всех системных компонентах в определенном образце набора. Типы свидетельства, необходимые для поддержки процесса оценки, включают в себя любое или все следующее:

**Коллекция свидетельств**

* Начальная документация, выделенная в разделе ["Начальная отправка документации](#initial-document-submission) " выше 
* Документы политики 
* Обработка документов 
* Параметры конфигурации системы 
* Изменение билетов 
* Изменение записей элементов управления 
* Системные отчеты

Для сбора свидетельства, необходимого для завершения процесса оценки, будут использоваться различные методы.  Эта коллекция свидетельств может иметь следующий вид: 
* Документы 
* Снимки экрана 
* Интервью 
* Общий доступ к экранам 

Используемые методы сбора свидетельств будут определены в процессе оценки. Конкретные примеры типа свидетельства, необходимого для отправки, см. в руководстве [по образцу свидетельства](../docs/certification-sample-evidence-guide.md).

**Действия по оценке**

Аналитики сертификации проверят свидетельство, которое вы предоставляете, чтобы определить, выполнены ли соответствующие требования в рамках этой спецификации сертификации Microsoft 365. 

По возможности и чтобы сократить время, необходимое для завершения оценки, следует заранее укажите любую или всю документацию, [](#initial-document-submission) описанную в первоначальной отправке документации.

Сначала аналитики сертификации проверят свидетельство, предоставленное в первоначальной отправке документации, и сведения об аттестации издателя, чтобы определить соответствующие строки запроса, размер выборки и необходимость получения дополнительных свидетельств, как описано выше.  Аналитики сертификации проанализирует всю собранную информацию, чтобы сделать выводы о том, как и если вы выполняете элементы управления в этой спецификации сертификации Microsoft 365. 

## <a name="app-certification-criteria"></a>Критерии сертификации приложений

Ваше приложение, вспомогательная инфраструктура и вспомогательная документация будут оцениваться в следующих доменах безопасности:

1. [**Application Security**](#application-security)
1. [**Операционная безопасность и безопасное развертывание**](#operational-security)
1. [**Безопасность и конфиденциальность обработки данных**](#data-handling-security-and-privacy)

Каждый из этих доменов безопасности включает определенные ключевые элементы управления, охватывающие одно или несколько конкретных требований, которые будут оцениваться в рамках процесса оценки. Чтобы убедиться, что сертификация Microsoft 365 является инклюзивной для разработчиков всех размеров, каждый из доменов безопасности оценивается с помощью системы оценки, чтобы определить общую оценку по каждому из доменов. Оценки для каждого из элементов управления сертификацией Microsoft 365 выделяются в диапазоне от 1 (низкий) до 3 (высокий) в зависимости от предполагаемого риска того, что этот элемент управления не будет соблюдены. Каждый из доменов безопасности будет иметь минимальную процентную отметку, которая будет считаться проходом. Некоторые элементы этой спецификации включают в себя некоторые критерии автоматического сбоя:

- Разрешения API, не соответствующие принципу наименьших привилегий (PoLP).  
- Отчет о тестировании на проникновение при необходимости не создается.
- Отсутствие защиты от вредоносных программ
- Многофакторная проверка подлинности не используется для защиты административного доступа.  
- Процессы исправления отсутствуют.  
- Нет подходящего [уведомления о конфиденциальности GDPR](#gdpr) .  

## <a name="application-security"></a>Application Security

Домен безопасности приложений посвящен следующим трем областям: 
* Проверка разрешений GraphAPI 
* Проверки внешнего подключения
* Тестирование безопасности приложений 

### <a name="graphapi-permission-validation"></a>Проверка разрешений GraphAPI

Проверка разрешений GraphAPI выполняется, чтобы убедиться, что приложение или надстройка не запрашивают разрешения с превышением разрешений. Для этого вручную проверяется, какие разрешения запрашиваются. Аналитики сертификации будут перекрестно ссылаться на эти проверки в отправке аттестации издателя и оценивать запрашиваемый уровень доступа, чтобы обеспечить соответствие методикам минимальных привилегий. Если аналитики сертификации считают, что эти методы минимальных привилегий не выполняются, аналитики сертификации будут иметь открытое обсуждение для проверки бизнес-обоснования запрашиваемых разрешений. Любые несоответствия с отправкой аттестации издателя, обнаруженные во время этой проверки, также будут получать отзывы, чтобы можно было обновить аттестацию издателя. 

### <a name="external-connectivity-checks"></a>Проверки внешнего подключения

В рамках оценки аналитик выполнит упрощенное пошаговое руководство по функциональным возможностям приложений для определения подключений за пределами M365.  Все подключения, которые не определены как подключения Майкрософт или прямые подключения к службе, будут помечены и рассмотрены во время оценки.

### <a name="application-security-testing"></a>Тестирование безопасности приложений

Достаточное рассмотрение рисков, связанных с вашим приложением или надстройки и вспомогательной средой, важно для предоставления клиентам гарантии безопасности приложения или надстройки. Тестирование безопасности приложений в форме тестирования на проникновение ДОЛЖНО выполняться, если приложение имеет подключение к любой службе, не опубликованной корпорацией Майкрософт. Если приложение работает автономно без подключения к какой-либо службе или серверной части сторонних поставщиков, тестирование на проникновение не требуется.


**Область тестирования на проникновение**

Действия по тестированию на проникновение должны выполняться в рабочей среде, которая поддерживает развертывание приложения или надстройки (например, где размещается код приложения или надстройки, который обычно является ресурсом в файле манифеста) вместе с любой дополнительной средой, которая поддерживает работу приложения или надстройки (например, если приложение или надстройка взаимодействует с другими веб-приложениями за пределами Microsoft 365).  При определении области необходимо быть внимательным, чтобы все "подключенные" системы или среды, которые могут повлиять на безопасность среды в области, также были включены в все действия по тестированию на проникновение. 

Если методы используются для сегментирования сред в области от других сред, действия по тестированию на проникновение ДОЛЖНЫ проверять эффективность указанных методов сегментации. Это должно быть подробно описано в отчете о тестировании на проникновение. 

Отчеты о тестировании на проникновение будут проверяться, чтобы убедиться в отсутствии уязвимостей ****, соответствующих следующим условиям автоматического сбоя, описанным в элементах управления ниже.
 
**Требования к тестированию на проникновение**


|**Тип условия**|**Элементы управления тестами на проникновение**|
| -------------------------|-----------------------------|
|**Общие критерии**| **Controls**|
|| Тестирование на проникновение приложений и **инфраструктуры должно** проводиться ежегодно (каждые 12 месяцев) и проводиться независимой компанией, которая является известной. |

**Требования и правила для дополнительного тестирования на проникновение**

- Для независимых поставщиков программного обеспечения, которые в настоящее время не задействованы в тестировании на проникновение, тестирование на проникновение можно проводить бесплатно с помощью сертификации Microsoft 365. Корпорация Майкрософт упорядочивает и окупит затраты на тест на проникновение в течение 12 дней ручного тестирования. Затраты на тесты на проникновение вычисляются на основе количества дней, необходимых для тестирования среды. За все расходы, которые превышают 12 дней тестирования, отвечает независимый поставщик программного обеспечения. 
- Независимые поставщики программного обеспечения должны будут отправлять свидетельство и получать утверждение для 50 % элементов управления в области до того, как будет проводиться тест на проникновение. Чтобы приступить к работе, просто заполните начальную отправку документа и выберите тест на проникновение, включенный в оценку. Когда вы завершите 50 % элементов управления, вам будет предложено задать область и запланировать тест на проникновение.
- Отчет, выданный после завершения проверки пера, будет предоставлен независимому поставщику программного обеспечения после завершения сертификации. Этот отчет вместе с сертификацией Microsoft 365 можно использовать, чтобы показать потенциальным клиентам, что ваша среда защищена.
- Независимые поставщики программного обеспечения также будут отвечать за демонстрацию того, что уязвимости, обнаруженные в тесте на проникновение, были устранены до получения сертификации, но не нужно создавать чистый отчет.

После упорядочения теста на проникновение независимый поставщик программного обеспечения отвечает за плату, связанную с планированием и отменой, следующим образом:

## <a name="optional-external-compliance-frameworks-review"></a>Проверка необязательных внешних платформ соответствия требованиям

Хотя это не является обязательным, если вы в настоящее время соответствуете стандарту ISO 27001, PCI DSS или SOC2, вы можете использовать эти сертификаты для удовлетворения некоторых элементов управления сертификацией Microsoft 365. Аналитики сертификации попытаются согласовать существующие внешние платформы безопасности со спецификацией сертификации Microsoft 365. Однако если вспомогательная документация не может предоставить гарантии того, что элементы управления сертификацией Microsoft 365 были оценены как часть аудита и оценки внешних платформ безопасности, необходимо предоставить дополнительные доказательства того, что указанные элементы управления уже созданы.

Документация должна надлежащим образом продемонстрировать, что среда в области сертификации Microsoft 365 была включена в область этих внешних платформ безопасности. Проверка этих платформ безопасности будет выполнена путем принятия свидетельства действительных сертификатов, выполненных известными внешними сторонними компаниями. Эти надежные компании должны быть членами международных органов по аккредитации для соответствующих программ соответствия требованиям. Ознакомьтесь со стандартами сертификации и соответствия стандартов ISO для ISO 27001 и квалифицированных оценщиков безопасности (QSA) для PCI DSS.

В следующей таблице выделены внешние платформы и документация, необходимые аналитикам сертификации в рамках этого процесса проверки.

| **Standard** | **Требования** |
| ----- | ----- |
| **[ISO 27001](#iso-27001)** | Потребуется общедоступная версия заявления о применимости **(** SOA) и копия выданного сертификата ISO 27001.  SoA суммирует вашу позицию по каждому из 114 элементов управления информационной безопасностью и будет использоваться для определения исключения элементов управления, которые не удовлетворительно описаны в сертификате ISO 27001. Если это невозможно определить, просмотрев общедоступную версию SOA, аналитику может потребоваться доступ к полному SOA, если ISO 27001 будет использоваться для проверки некоторых элементов управления спецификацией сертификации Microsoft 365.  Помимо проверки области действий по оценке ISO 27001, аналитики также подтвердит действительность компании аудита, как описано выше.|
|**[PCI DSS](#pci-dss)**| Должен быть **предоставлен** допустимый документ аттестации соответствия уровня 1 (AOC), четко определяющий приложения и системные компоненты в области действия.  AOC самооценки  не будет приниматься в качестве свидетельства рекомендаций по обеспечению безопасности собраний. ### <a name="tls-profile-configuration-requirements"></a>Требования к конфигурации профиля TLS

Весь сетевой трафик, будь то в виртуальной сети, облачной службе или центре обработки данных, должен быть защищен как минимум TLS версии 1.1 (рекомендуется TLS версии 1.2+ ) или другим применимым протоколом. Исключения из этого требования:

* **Перенаправление http-to-HTTPS**. Ваше приложение может отвечать по протоколу HTTP на перенаправление клиентов по протоколу HTTPS, но ответ не должен содержать конфиденциальные данные (файлы cookie, заголовки, содержимое). Другие HTTP-ответы, кроме перенаправления по протоколу HTTPS и реагирования на пробы работоспособности, не допускаются. См. ниже.
* **Пробы работоспособности**. Ваше приложение может отвечать на пробы работоспособности по протоколу HTTP **,** только если проверки работоспособности HTTPS не поддерживаются проверяющей стороной.
* **Доступ к сертификату**. Доступ к конечным точкам CRL, OCSP и AIA для проверки сертификатов и отзыва разрешен по протоколу HTTP.
* **Локальные коммуникации**. Ваше приложение может использовать ПРОТОКОЛ HTTP (или другие незащитимые протоколы) для обмена данными, которые не покидают операционную систему, например. ж. подключение к конечной точке веб-сервера, предоставляемой на localhost.

Сжатие TLS **ДОЛЖНО** быть отключено.

### <a name="encryption-profile-configuration-requirements"></a>Требования к конфигурации профиля шифрования

Только криптографические примитивы и параметры допускаются следующим образом:

### <a name="symmetric-cryptography"></a>Симметричное шифрование

**Шифрование**

&emsp;&#x2713; разрешены только AES, BitLocker,Как или TDES. Допускается любая из поддерживаемых >=128 (128, 192 и 256 бит) и может использоваться (рекомендуется использовать 256-разрядные ключи).

&emsp;&#x2713; разрешен только режим CBC. Каждая операция шифрования должна использовать новый, случайно созданный вектор инициализации (IV).

&emsp;&#x2713; использовать потоковые шифры, такие как RC4, **НЕ разрешено** .

**Хэш-функции**

&emsp;&#x2713; весь новый код должен использовать SHA-256, SHA-384 или SHA-512 (в совокупности называется SHA-2). Выходные данные могут быть усечены до не менее 128 бит

&emsp;&#x2713; SHA-1 можно использовать только в целях совместимости.

&emsp;&#x2713; использование MD5, MD4, MD2 и других хэш-функций ЗАПРЕЩЕНО даже для приложений, не использующих шифрование.

**Проверка подлинности сообщений**

&emsp;&#x2713; весь новый код должен использовать HMAC с одной из утвержденных хэш-функций. Выходные данные HMAC могут быть усечены до не менее 128 бит.

&emsp;&#x2713; HMAC-SHA1 можно использовать только в целях совместимости.

&emsp;&#x2713; HMAC должен быть не менее 128 бит. Рекомендуется использовать 256-разрядные ключи.

### <a name="asymmetric-algorithms"></a>Асимметричные алгоритмы

**Шифрование**

&emsp;&#x2713; RSA разрешено. Ключ **ДОЛЖЕН** содержать не менее 2048 бит, и необходимо использовать заполнение OAEP. Использование заполнения PKCS разрешено только из соображений совместимости.

**Подписи**

&emsp;&#x2713; RSA разрешено. Ключ **ДОЛЖЕН** содержать не менее 2048 бит, и необходимо использовать заполнение PSS. Использование заполнения PKCS разрешено только из соображений совместимости.

&emsp;&#x2713;ECDSA разрешено. Ключ **должен** содержать не менее 256 бит. Необходимо использовать кривую NIST P-256, P-384 или P-521.

**Обмен ключами**

&emsp;&#x2713; ECDH разрешено. Ключ **должен** содержать не менее 256 бит. Необходимо использовать кривую NIST P-256, P-384 или P-521.

&emsp;&#x2713; ECDH разрешено. Ключ **должен** содержать не менее 256 бит. Необходимо использовать кривую NIST P-256, P-384 или P-521.

Все требования, отсутствующие в ISO 27001, необходимо включить в действия по оценке сертификации Microsoft 365.

**Защита от вредоносных программ — антивирусная программа**

Если защита от вредоносных программ выполняется с помощью управления приложениями и подтверждается в отчете ISO 27001, дальнейшее исследование не требуется. Если управление приложениями не установлено, аналитикам сертификации потребуется определить и оценить признаки механизмов управления приложениями, чтобы предотвратить отключение вредоносных программ в среде. Для этого потребуется:

* Продемонстрируйте, что антивирусное программное обеспечение работает во всех примерах системных компонентов.

* Продемонстрировать, что антивирусная программа настроена для всех компонентов выборки системы для автоматической блокировки вредоносных программ, помещения в карантин & оповещения или оповещения.

* Антивирусное программное обеспечение **должно быть** настроено для записи всех действий в журнал.

**Управление исправлениями — исправление**

Так как аудиты ISO 27001 не оценивают эту категорию специально, вам потребуется:

* Программные компоненты и операционные системы, которые больше не поддерживаются поставщиком, **не** должны использоваться в среде. Должны быть настроены политики поддержки, чтобы неподдерживаемые программные компоненты и операционные системы были удалены из среды, и процесс, определяющий, когда компоненты программного обеспечения должны завершать жизненный цикл.

**Сканирование уязвимостей**  

Так как аудиты ISO 27001 не оценивают эту категорию специально, вам потребуется:

* Покажите, что проводится ежеквартальное внутреннее и внешнее сканирование уязвимостей.

* Убедитесь, что имеется вспомогательная документация по исправлению уязвимостей на основе ранжирования рисков и в соответствии со спецификацией следующим образом:
 
 &#x2713; все критические и высокие риски в рамках ранжирования риска для внутреннего сканирования.
 
 &#x2713; все критические, высокоуровневые и средние проблемы риска в рамках ранжирования риска для внешнего сканирования.
 
 &#x2713;, что исправление выполняется в согласованном режиме с задокументированной политикой исправления уязвимостей.

**Брандмауэр — брандмауэры (или эквивалентные технологии)**

Так как аудиты ISO 27001 не оценивают эту категорию специально, вам потребуется:

* Продемонстрировать, что брандмауэры устанавливаются на границе среды в области действия.

* Продемонстрировать, что брандмауэры устанавливаются между сетью периметра и доверенными сетями.

*   Покажите, что весь открытый доступ завершается в сети периметра.

*   Продемонстрировать, что учетные данные администратора по умолчанию изменяются перед установкой в динамическую среду.

*   Продемонстрировать, что весь разрешенный трафик через брандмауэры проходит через процесс авторизации, что приводит к документации по всем трафику с бизнес-обоснованием.

*   Продемонстрировать, что все брандмауэры настроены для удаления трафика, не определенного явным образом.

*   Продемонстрировать, что брандмауэры поддерживают только надежное шифрование во всех административных интерфейсах, не включающих консоль.

*   Продемонстрировать, что неконсолятивные административные интерфейсы брандмауэра, предоставляемые через Интернет, поддерживают MFA.

*   Продемонстрировать, что проверки правил брандмауэра выполняются по крайней мере каждые 6 месяцев.

**Брандмауэр — брандмауэры веб-приложений (WAF)**  

При развертывании WAF будет предоставлен дополнительный кредит для защиты от множества угроз и уязвимостей веб-приложений, к которых может быть предоставлено приложение. При наличии WAF или аналогичного шаблона вам потребуется:

* Продемонстрировать, что WAF настроен в активном режиме защиты или отслеживает больше с помощью оповещений.

* Продемонстрировать, что WAF настроен для поддержки разгрузки SSL.

* Настраивается согласно основному набору правил OWASP (3.0 или 3.1) для защиты от большинства из следующих типов атак:

&#x2713; проблемы с протоколом и кодированием.

&#x2713; внедрения заголовка, отправки запросов и разделения ответов.

&#x2713; атаки обхода файлов и путей.

&#x2713; удаленного включения файлов (RFI).

&#x2713; атаки удаленного выполнения кода.

&#x2713; атак путем внедрения PHP.

&#x2713; межсайтовых сценариев.

&#x2713; атак путем внедрения кода SQL.

&#x2713; атак с фиксацией сеансов.

**Элемент управления изменениями**

Так как аудиты ISO 27001 не оценивают некоторые элементы процессов запроса на изменение, вам потребуется:

* Продемонстрировать, что запросы на изменение содержат следующие сведения:

&#x2713; задокументированное влияние.

&#x2713; сведения о тестировании функциональных возможностей.

&#x2713; сведения о любых процедурах резервного копирования.

* Продемонстрировать, что тестирование функциональных возможностей проводится после завершения изменений.

* Показано, что запросы на изменение отключаются после тестирования функциональных возможностей.

**Управление учетными записями**

Так как аудиты ISO 27001 не оценивают некоторые элементы процессов управления учетными записями, вам потребуется:

*   Покажите, &#x2713;реализованы для устранения атак воспроизведения (например, MFA, Kerberos).
*   Демонстрация отключения или удаления учетных записей, которые не использовались в течение 3 месяцев.
*   &#x2713; или другие подходящие способы устранения рисков должны быть настроены для защиты учетных данных пользователя. В качестве рекомендации следует использовать следующую минимальную политику паролей:

&#x2713; минимальная длина пароля — 8 символов.

&#x2713; блокировки учетной записи не более 10 попыток.
 
&#x2713; паролей не менее пяти паролей.
 
&#x2713; принудительное применение надежных паролей.
 
*   Продемонстрировать, что MFA настроена для всех решений для удаленного доступа.

Так как аудиты ISO 27001 не оценивают некоторые элементы этой категории, вам потребуется:

### <a name="hosting-deployment-types"></a>Типы развертывания размещения

Корпорация Майкрософт подтверждает, что вы развернете приложения и сохраните код приложения или надстройки в разных средах размещения. Приложение F анализирует распространенные типы развертывания и сопоставляет их с элементами управления безопасностью, которые оцениваются в процессе оценки. Определены следующие типы развертывания размещения:

|Типы размещения  |Описание  |
|-----|------|
|**ISV Hosted**|Размещаемые типы независимых поставщиков программного обеспечения можно определить как место, где вы отвечаете за инфраструктуру, используемую для поддержки среды приложения или надстройки. Он может физически находиться в собственных центрах обработки данных или сторонних центрах обработки данных с помощью службы совместного размещения. В конечном счете вы имеете полное владение и административный контроль над вспомогательной инфраструктурой и операционной средой.|
|**Инфраструктура как услуга (IaaS)** (https://azure.microsoft.com/overview/what-is-iaas/)|Инфраструктура как услуга — это предоставляемая служба, которая управляется физической вспомогательной инфраструктурой и обслуживается от их имени поставщиком облачных служб (CSP). Как правило, за сеть, хранилище, физические серверы и инфраструктуру виртуализации отвечает CSP. Вы отвечаете за операционную систему, ПО промежуточного слоя, среду выполнения, данные и приложения. Возможности брандмауэра также будут управляться и поддерживаться сторонними поставщиками, однако обслуживание базы правил брандмауэра обычно по-прежнему будет отвечать за потребителей.|
|**Платформа как услуга или бессерверная (PaaS)** (https://azure.microsoft.com/overview/what-is-paas/)| С помощью платформы как услуги вы подготавливается с помощью управляемой платформы, которая представляет службу, которую можно использовать. Вам не нужно выполнять функции системного администратора, так как операционная система и вспомогательная инфраструктура управляются CSP. Обычно это используется, когда организации не хотят представлять веб-службу и вместо этого могут сосредоточиться на создании исходного кода веб-приложения и публикации веб-приложения в облачных управляемых веб-службах.  Другим примером может быть служба базы данных, в которой обеспечивается подключение к базе данных, однако вспомогательная инфраструктура и приложение базы данных абстрагируются от потребителя.   **Примечание. Бессерверные и PaaS похожи, поэтому для бессерверных типов развертывания размещения сертификации Microsoft 365 и PasS считаются одинаковыми.**|
|**Гибридное размещение**|В гибридном размещенном типе можно использовать несколько размещенных типов для поддержки различных частей вспомогательной среды. Это может быть более случай, когда приложения и надстройки используются в нескольких стеках M365. Несмотря на то, что сертификация Microsoft 365 поддерживает разработку приложений и надстроек в нескольких службах M365, оценка всей вспомогательной среды (между приложениями и надстройкими) должна оцениваться в соответствии с каждым из применимых сопоставлений размещенных типов. Иногда для одной надстройки могут использоваться различные размещенные типы. В этом случае применимость критериев по-прежнему будет соответствовать критериям "Сопоставления размещенных типов" для разных размещенных типов.|
|**Совместное размещение**|Общее размещение — это место, где вы размещаете среду на платформе, к которой имеют доступ несколько отдельных потребителей. Спецификация сертификации Microsoft 365 не была записана для учетной записи из-за внедрения облака. Совместное размещение не является распространенным. Если вы считаете, что это используется, обратитесь в корпорацию Майкрософт, так как необходимо создать дополнительные требования для учета дополнительных рисков, связанных с этим типом размещения.|

## <a name="glossary"></a>Глоссарий

### <a name="aia"></a>AIA

*Доступ к данным центра — это дескриптор расположения службы, используемый для поиска сертификата выдающего центра сертификации.

### <a name="crl"></a>CRL

*Список отзыва сертификатов предоставляет средства для конечной точки SSL, чтобы убедиться, что сертификат, полученный от удаленного узла, является допустимым и надежным.

### <a name="cvss-score"></a>Оценка CVSS

*Общая система оценки уязвимостей — это опубликованный стандарт, который измеряет уязвимость и вычисляет числовую оценку на основе ее серьезности.

### <a name="cvss-patch-management-guidelines"></a>Рекомендации по управлению исправлениями CVSS

* Критическое значение (от 9.0 до 10.0)
* High (от 7,0 до 8,9)
* Средний (от 4.0 до 6.9)
* Низкий (от 0,0 до 3,9)

### <a name="dmz"></a>DMZ

*Демилитаризованная зона — это физическая или логическая промежуточная сеть, которая взаимодействует непосредственно с внешними или несвойными сетями, сохраняя внутреннюю частную сеть узла отдельной и изолированной.

### <a name="euii"></a>EUII

*Идентифицируемые сведения для конечных пользователей*.

### <a name="gdpr"></a>GDPR

*Общий регламент по защите данных — это регламент ес о конфиденциальности и защите данных для всех граждан ЕС, независимо от того, где находится сайт приложения.

### <a name="hsts"></a>HSTS

*Http Strict Transport Security использует заголовок http-ответа, который указывает веб-браузеру получать доступ только к содержимому по протоколу HTTPS.  Она предназначена для защиты от атак на более раннюю версию и перехвата файлов cookie.

### <a name="iec"></a>IEC

*InternationalTechnical Commission.

### <a name="isms"></a>ISMS

*Система управления информационной безопасностью.

### <a name="isv"></a>Независимый поставщик программного обеспечения

Независимые поставщики безопасности — это частные лица и организации, которые разрабатывают, продают и продают программное обеспечение, которое работает на сторонних платформах программного и аппаратного обеспечения.

### <a name="iso-27001"></a>ISO 27001

Платформа спецификаций системы управления информационной безопасностью для всех технических элементов управления в процессах управления рисками и политик управления рисками в организациях.

### <a name="lfi"></a>LFI

*Включение локальных файлов* позволяет злоумышленнику включать файлы на сервер через веб-браузер.

### <a name="nist"></a>NIST

Национальный *институт* стандартов (NIST), не агентизованный орган Министерства торговли США, предоставляет организациям частного сектора в США рекомендации по оценке и утверждению их способности предотвращать, обнаруживать кибератаки и реагировать на них.

### <a name="non-significant-changes"></a>Несуществуют изменения

* Исправления незначительных ошибок.
* Незначительные улучшения производительности.
* Операционные системы, библиотеки, исправления клиентских и серверных приложений.

### <a name="ocsp"></a>OCSP

*Протокол состояния сертификата* в сети используется для проверки состояния отзыва цифровых сертификатов X.509.

### <a name="oii"></a>OII

*Идентифицируемая информация организации*.

### <a name="owasp"></a>OWASP

*Откройте проект безопасности веб-приложений*.

### <a name="pci-dss"></a>PCI DSS

*Стандарт безопасности данных для отрасли* платежных карт— организация, которая поддерживает стандарты безопасности данных владельцев карт по всему миру.

### <a name="pen-testing"></a>Тестирование пера

*Тестирование на проникновение* — это метод тестирования веб-приложения путем имитации вредоносных атак для поиска уязвимостей безопасности, которые злоумышленник может использовать.

### <a name="saml"></a>SAML

*Язык разметки утверждений* безопасности — это открытый стандарт для обмена данными проверки подлинности и авторизации между пользователем, поставщиком удостоверений и поставщиком услуг.

### <a name="sensitive-data"></a>Конфиденциальные данные

* Данные управления доступом.
* Содержимое клиента.
* Сведения об удостоверении конечного пользователя.
* Данные поддержки.
* Общедоступные персональные данные.
* Анонимная информация конечного пользователя.

### <a name="significant-changes"></a>Значительные изменения

* Перемещение среды размещения.
* Основные обновления вспомогательной инфраструктуры; Например, реализация нового брандмауэра, основные обновления до служб переднего плана и т. д.
* Добавление возможностей и (или) расширений в приложение.
* Актуальные данные приложение, которое записывает дополнительные конфиденциальные данные.
* Изменения в потоках данных или моделях авторизации приложения
* Добавление конечных точек API или функций конечной точки API.

### <a name="soc-2"></a>SOC 2;

*Элемент управления 2* организации-службы— процедура технического аудита, состоящая из пяти принципов управления безопасностью, чтобы поставщики услуг безопасно управляют данными и конфиденциальностью клиентов организации.

### <a name="ssl"></a>SSL

*Безопасный уровень сокетов*.

### <a name="tls"></a>TLS;

*Безопасность транспортного уровня*.
